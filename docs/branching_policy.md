# Branching Policy
본 프로젝트는 GitFlow의 단순화된 변형을 사용한다.
브랜치는 "파일 종류"가 아니라 "개발 단계와 통합 흐름"을 기준으로 구분한다.

---

## Branch Roles

### main
- 항상 **빌드 및 실행 가능한 안정 상태**를 유지
- 외부에 보여줄 수 있는 대표 버전
- 일정 수준의 완성도에 도달했을 때만 dev에서 병합
- main 병합 시 태그를 생성 (ex. v0.1-core-stable)

### dev
- **중간 통합 브랜치(integration branch)** 역할
- 여러 작업 브랜치(feature/refactor)가 병합되는 지점
- 다음 릴리즈 후보를 검증하는 용도로 사용
- 기능은 미완일 수 있으나, 기본적인 플레이/빌드는 가능해야 함

### feature/\*, refactor/\*
- 실제 개발 작업은 이 브랜치들에서 수행
- 코드 변경과 관련 문서/일지는 **같은 브랜치에서 함께 수정**
- 커밋은 목적 단위로 분리
  - ex. 코드 변경 커밋 + 문서/일지 커밋

---

## Workflow

1. main 또는 dev에서 작업 브랜치를 생성
2. 작업 브랜치에서 기능/리팩터링/문서 작업을 수행
3. 작업이 하나의 의미 있는 단위로 완료되면 dev로 병합
4. dev가 충분히 안정화되면 main으로 병합
5. main 병합 직후 태그를 생성해 기준점을 남김

---

## Commit Guidelines
- 커밋 메시지는 변경 목적이 드러나도록 작성
  - feat: 새로운 기능
  - refactor: 구조 개선
  - docs: 문서/일지 변경
- 하나의 커밋에는 하나의 목적만 포함
